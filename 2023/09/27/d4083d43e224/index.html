Spring-IoC的认识

发表于2023-09-27|更新于2023-09-27|Spring框架的学习

Spring的编程模型

  • 面向对象编程
    • 契约接口：Aware、BeanPostprocessor …
    • 设计模式：观察者模式、组合模式、模板模式…
    • 对象继承：Abstract*类
  • 面向切面编程
    • 动态代理：JdkDynamicAopProxy
    • 字节码提升：ASM、CGLib、AspectJ…
  • 面向元编程
    • 注解：模式注解（@component、@Service、@Respository… ）
    • 配置：Environment抽象、PropertySources、BeanDefinition…
    • 泛型：GenericTyperesolver、ResolvableType…
  • 函数驱动
    • 函数接口：ApplicationEventPublisher
    • Reactive：Spring WebFlux
  • 模块驱动
    • Maven Artifacts
    • OSGI Bundles
    • Java 9 Automatic Modules
    • Spring @Enable*

IoC发展简介

什么是IoC？
在软件工程中，IOC（控制反转） 是一种设计原则，它指的是将控制流程的权力从应用程序代码中转移到框架或容器中。这样做的目的是提高代码的可维护性和可扩展性，同时降低组件之间的耦合度。常见的 IOC 实现方式包括依赖注入

IoC主要实现策略

  • 服务定位模式（是JavaEE里面定义的一种模式，通常通过JNDI这种技术获取JAVAEE的组件，比如获取EJB组件或者DataSource这种东西）
  • 依赖注入 ：
    • 构造器注入
    • 参数注入
    • Setter注入
    • 接口注入
  • 上下文的依赖查询（比如Java里面的JavaBean的技术，它里面有一个通用的上下文beancontext，既可以传输Bean，也可以管理Bean的层次性）
  • 模板方法的设计模式（Spring JDBC里面的JDBC Template）
  • 策略模式

IoC容器的职责

IoC主要服从以下几个设计的目的：

  • 实现与执行的任务之间要产生解耦
  • 要关注这个任务上的模块它所赋予设计的一个目的（也就是说关注最终的目标，而不是具体的实现）
  • 释放这个模块，系统知道它是怎么运作的，但是不需要依赖某个契约（比如JavaEE中的JNDI可以帮助我们把组件的实现和组件之间产生解耦，但是还是依赖摸个契约，即依赖JNDI lookup查询接口来进行查询）
  • 当模块取消的时候预防副作用

IoC通常会被用作另外一种想法：好莱坞原则：不要来找我们，我们回来找你

通用的职责：

  • 依赖处理
    • 依赖查找
    • 依赖注入
  • 生命周期管理
    • 容器
    • 托管的资源（Java Beans或其他资源）
  • 配置
    • 容器
    • 外部化配置
    • 托管的资源（Java Beans或其他资源）

传统IoC容器实现

Java Beans作为IoC容器，所具有的的特性：

  • 依赖查找
  • 生命周期管理
  • 配置元信息
  • 事件
  • 自定义
  • 资源管理
  • 持久化

依赖查找 VS 依赖注入

优劣对比：

类型 依赖处理 实现便利性 代码侵入性 API依赖性 可读性
依赖查找 主动获取 相对繁琐 侵入业务逻辑 依赖容器API 良好
依赖注入 被动提供 相对遍历 低侵入性 不依赖容器API 一般

面试题精选（针对本节）

什么是IoC？

答：简单的说，IoC是反转控制，类似于好莱坞原则，主要有依赖查找和依赖注入实现

Spring Framework有哪些核心模块？

答：

  • spring-core：Spring基础API模块，如资源管理、泛型处理
  • spring-beans：Spring Bean相关，如依赖查找、依赖注入
  • spring-aop：Spring AOP处理，如动态处理、AOP字节码提升
  • spring-context：事件驱动、注解驱动、模块驱动等
  • spring-expression：Spring表达式语言模块

依赖查找和依赖注入的区别？

答：依赖查找是主动或手动的依赖查找方式，通常需要依赖容器或标准API实现；而依赖注入则是手动或自动依赖绑定的方式，无需依赖特定的容器和标准API

依赖查找和依赖注入的区别？

答：（笼统的回答）

  • 典型的IoC管理，依赖查找和依赖注入
  • AOP抽象
  • 事务抽象
  • 事件机制
  • SPI扩展
  • 强大的第三方整合
  • 易测试性
  • 更好的面向对象
